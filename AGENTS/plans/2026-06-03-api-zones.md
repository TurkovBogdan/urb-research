## Цель

HTTP-поверхность разделена на три явные зоны — `internal` (SPA + auth), `api` (внешний токен),
`webhook` (входящие). Защита **включена по умолчанию на всю зону** (default-on), а метка на маршруте
**снимает умолчание** для конкретного endpoint'а (`@guard("public")` — снять auth). Защита — это
**FastAPI-зависимости (guard'ы)**: на одном маршруте их может быть несколько (например «залогинен» +
«хватает уровня прав»), выполняются по порядку.

Права — модель **CASL**: правило = код `action:subject` (`admin:read`, `manage:all`); маршрут несёт
требуемый код меткой, проверку делает `core_users` против способностей пользователя
(`user.abilities`, уже есть у мок-юзера). **Ядро словарь прав НЕ знает** — метка несёт код как
непрозрачную строку (фиксировать перечень меток/прав в ядре нельзя, система должна быть гибкой).

Распределение ответственности — целевое состояние:
- **`core_users`** (модуль) — владеет реальной логикой: guard аутентификации (залогинен?) и guard
  прав (CASL: читает коды-метки маршрута → сверяет с `user.abilities`), модели User/Session/правил.
  **Регистрирует свои guard'ы в общем реестре ядра** при регистрации модуля (`register_guards(registry)`:
  `add("auth", …)`, `add("ability", …)`). Реестр не зонный — guard работает в любой зоне.
- **Ядро** — задаёт три зоны (архитектурные, не права), даёт **один явный декоратор** `@guard("вид",
  *аргументы)`, который навешивает на маршрут **непрозрачные данные** (`@guard("ability", "admin:read")` —
  код права; ядро его не интерпретирует), и держит **реестр** `вид → guard`, куда модули регистрируют
  свои guard'ы. `@guard("вид")` ссылается на guard по имени; `create_app` **валидирует**, что все виды
  зарегистрированы (иначе ошибка на старте). В ядре предрегистрированы **два универсальных guard'а** (без
  auth-логики): `public` («всё разрешает») и `disabled` («всё кладёт»; он же фолбэк → secure-by-default).
  Своей auth/прав-логики не содержит.
- **Прочие модули** — только помечают маршруты декоратором `@guard(...)` **из ядра** (любой вид/код).
  От `core_users` не зависят.

## Контекст

Сейчас всё под `/api/<module>` и по факту является internal (SPA): 10 модульных роутеров + `/api/core` +
settings; 9 фронтовых `features/*/api.ts` с `BASE='/api/...'`. Нет внешнего токенного API, нет вебхуков,
нет auth (есть мок `current_user`). Стандарт ошибок `{error, code?, fields?}` уже в `src/core/api/`
(см. `AGENTS/memory/core_api_standards.md`). FastAPI-терминология (research 2026): у FastAPI нет «guard»
как сущности — есть **middleware** (ASGI, до роутинга, на каждый запрос) и **dependencies** (`Depends`,
после роутинга, видят маршрут, прерываются `raise`). Наш «guard» = dependency; «middleware» = сквозная
ASGI-логика. Дизайн: `AGENTS/docs/api-zones.md`; схема auth: `AGENTS/research/fastapi-auth/INDEX.md`.

Research (2026-06-04, urb-research ×3) подтвердил архитектуру: (1) auth — это **зависимости, не
middleware** (гранулярность, видимость маршрута, OpenAPI, `raise` для прерывания); middleware оставляем
сквозному (логирование/CORS/rate-limit/request-id). (2) Монтаж зоны через
`app.include_router(zone, dependencies=[Depends(guard)])` — идиоматичен. (3) Несколько guard'ов на
маршруте складываются и выполняются по порядку; `Security(dep, scopes=[...])` — штатный способ для
scope внешней зоны `api` с отражением в OpenAPI. (4) `request.state.user` — рабочий паттерн (нюанс
конкуренции → альтернатива `contextvars`). (5) По «public» research показал две идиомы (метка-opt-out vs
отдельный public-роутер); выбрана **метка** — ровно модель «default-on на зону + снять умолчание на
маршруте», которую сформулировал пользователь.

## Область

**В области:**
- `src/core/router/` — зон-роутеры (`internal`/`api`/`webhook`) + декоратор-метка `@guard` + `zone_guard`.
- `src/core/guards/` — `GuardRegistry` + встроенные `public`/`disabled`; `src/core/middleware/` — ASGI-middleware ядра.
- Контракт `Module`: под-роутеры `router_internal`/`router_api`/`router_webhook` **+ метод
  `register_guards(registry)`** (по умолчанию no-op) — шов регистрации guard'ов.
- `create_app`: агрегировать под-роутеры модулей, **собрать реестр** (предрегистрация `public`/`disabled`
  + `register_guards` каждого модуля), смонтировать **зону `internal`** с `zone_guard`; перенести
  эндпоинты ядра (tasks/locks) в зону internal.
- **Только зона `internal` в этом плане.** `api`/`webhook` пока **НЕ монтируются** (отключены) — их виды
  (`scope`/`signature`) и каркас придут позже. Файлы зон `api`/`webhook` не создаём сейчас.
- **Без `/health`** в этом плане (health отключён). При выключенном API процесс HTTP-маршрутов не отдаёт.
- **Мок-guard'ы `auth`/`ability` (в этом плане)** — чтобы зона работала end-to-end без `core_users`:
  `auth` всегда авторизует (кладёт `request.state.user = _MOCK_USER`), `ability` — админ с read/write на
  всех группах (`_MOCK_USER.abilities` ≈ `manage:all` ⇒ любой `@guard("ability",…)` проходит). Живут в
  существующем `src/core/api/auth.py` (где уже `_MOCK_USER`), регистрируются в `create_app` **временно**
  (помечены MOCK; снимаются, когда `core_users` зарегистрирует реальные).
- Дефолтный `disabled` (фолбэк), пока вид не зарегистрирован/у зоны нет умолчания (secure-by-default).
- **`BACKEND_API_ENABLED`** (env + CLI, **дефолт `false`**, CLI > env > default) — гейтит **монтаж
  API-поверхности** (зон-роутеры/эндпоинты). `false` ⇒ API-методы НЕ биндятся, приложение работает в
  режиме «только scheduler + выполнение задач» (DB/scheduler/tasks через lifespan идут как обычно).
  `true` ⇒ зона internal монтируется и обслуживается.
- Миграция `/api/<m>` → `/internal/<m>`: backend-монтаж, фронтовые BASE через общую обёртку, тесты.

**Следующий шаг (не в этом заходе):**
- **Валидация на сборке** (`_validate_guard_rules` + проверка умолчательных видов зоны): каждый вид в
  `@guard(...)` и в `default=[...]` зоны должен быть в реестре, иначе ошибка на старте. Делаем следующим
  шагом после каркаса.

**Вне области (отдельные заходы потом):**
- Модуль `core_users` сам по себе (модели User/Session/прав + **реальные** guard'ы `auth`/`ability`,
  заменяющие мок этого плана) — он лишь **потребитель** шва, реализуемого здесь.
- Реальные guard'ы внешнего API (`scope`) и вебхука (`signature`).
- Версионирование внешнего `/api/v1`.
- Поведение фронтовой обёртки по ошибкам/401 сверх владения базой `/internal`.

## Подход

**Зона = префиксное пространство роутеров** поверх общего реестра guard'ов. Кроме **префикса** зону
характеризует только её **умолчательный список видов** (default-on), задаваемый на монтаже
(`internal`→`["auth"]`, позже `api`→`["scope"]`, `webhook`→`["signature"]`). Сами guard'ы зону не знают —
реестр один на все зоны. Монтаж зоны:
`app.include_router(zone_router, prefix="/internal", dependencies=[Depends(zone_guard)])`.

Ядро держит по одному **агрегирующему** `APIRouter` на зону; модуль отдаёт свой под-роутер на зону, а
`create_app` включает под-роутер каждого модуля в соответствующий зон-агрегатор и монтирует три
агрегатора (`/internal`, `/api`, `/webhook`).

**Default-on по зоне + снятие умолчания.** Каждая зона монтируется с зон-guard'ом по умолчанию
(`app.include_router(zone, prefix=..., dependencies=[Depends(zone_guard)])`). `zone_guard` читает метки
endpoint'а: `public` ⇒ запускает универсальный `public` (всё разрешает); `disabled` ⇒ `disabled` (всё
кладёт); иначе — умолчание зоны + виды меток, каждый резолвится по реестру. Так «auth на все маршруты,
снять для login» — это умолчание зоны `auth` + метка `public`. (Реализуемо ТОЛЬКО зависимостями:
ASGI-middleware не видит endpoint до роутинга, снять per-route не может.)

**`public`/`disabled` — тоже guard'ы.** Не спец-случай в коде ядра, а два тривиальных универсальных
guard'а в `src/core/guards/`: `public` (`return` — всё разрешает), `disabled` (`raise` — всё кладёт).
Ядро предрегистрирует эти два вида в реестре и подставляет сами (auth-логики в них нет). `disabled`
вдобавок — **фолбэк**, когда у маршрута не набралось ни умолчания зоны, ни меток.

**Что считается «пусто» (когда `disabled`).** У маршрута нет списка исполнителей — только метки-данные
(`@guard(...)`); исполнители берутся из **реестра по виду** (+ умолчание зоны). Поэтому фолбэк `disabled`
срабатывает, когда у маршрута нет ни умолчания зоны, ни меток (не от «нет меток» самого по себе). Маршрут
без `@guard` при наличии умолчания — просто умолчание зоны (`auth`); `@guard("public")` — всё разрешает.
Метка вида, которого нет в реестре (есть `@guard("ability",…)`, но `core_users` не загружен) ⇒ **ошибка
на старте** (валидация), не молчаливое открытие. Итого: не-public маршрут, на котором не набралось ни
одного исполняющего guard, закрыт.

**Метки vs реализация (гибко, без словаря в ядре).** В ядре живёт **один декоратор** `@guard("вид",
*аргументы)`; он лишь **дописывает** на функцию непрозрачное правило `(вид, *аргументы)` — это просто
метаданные (декоратор НЕ зависимость; зависимость, читающая их, выполняется позже). Ядро вид/аргументы
НЕ парсит и перечень прав не фиксирует (исключение — два встроенных вида `public`/`disabled`, которые
ядро само регистрирует и исполняет). Прочие модули импортируют `@guard` из ядра без зависимости от
`core_users`. **Смысл** видов реализует `core_users`: его guard'ы читают правила endpoint'а
(`request.scope["endpoint"]`) и `user.abilities`. На защищённом маршруте guard'ы идут по порядку: сначала
умолчание зоны `auth` (кладёт `request.state.user`, иначе 401) → затем `ability` (берёт правила вида
`ability`, сверяет коды по CASL с `user.abilities`, иначе 403). Маршрут без `@guard` под умолчанием ⇒
только `auth`; `@guard("public")` ⇒ всё разрешает.

**Шов передачи guard'ов — общий реестр (ключевое решение).** В ядре есть **один общий реестр**
`вид → guard` (`GuardRegistry`) — **не разделённый по зонам**: зарегистрированный guard работает в любой
зоне. `core_users` — обычный `Module`; при регистрации модуля он **регистрирует свои guard'ы по имени**
(`register_guards(registry)`: `add("auth", …)`, `add("ability", …)`). Ядро предрегистрирует
`public`/`disabled`. `@guard("вид")` ссылается на guard **по имени**, а `create_app` после сбора всех
модулей **валидирует**: каждый вид на маршрутах должен быть в реестре, иначе — ошибка на старте (то самое
«guard должен быть зарегистрирован до использования»; проверка на сборке, т.к. декоратор отрабатывает
раньше — при импорте). Реестр строится один раз при композиции и дальше не мутируется. **Default-on
зоны** — это НЕ свойство реестра, а **параметр монтажа**: каждая зона включается с умолчательным списком
видов (`make_zone_guard(registry, default=["auth"])` для internal). Ядро фиксирует только **имена зон**
(архитектура), их умолчания и два встроенных вида `public`/`disabled`; всё про аутентификацию/права —
виды из `core_users`.

Guard'ы-зависимости лежат в `src/core/guards/`, сквозные ASGI-middleware — в `src/core/middleware/`
(терминологически разделены: middleware ≠ guard).

**Master-выключатель API — `BACKEND_API_ENABLED` (дефолт `false`).** Флаг (env + CLI, CLI > env >
default) гейтит **монтаж API-поверхности**: при `true` `create_app` монтирует зоны (`internal`/`api`/
`webhook`) с guard'ами и валидацией; при `false` зон-роутеры **не биндятся вовсе** — приложение
поднимается в режиме «только scheduler + выполнение задач» (lifespan: DB/миграции/старт модулей/scheduler
идут как обычно). `/health` остаётся всегда (инфра/пробы). Дефолт `false` ⇒ чтобы поднять API, его надо
включить явно. Это узкий флаг именно для API-биндинга; полноценное разделение процессов API/worker — в
parked-плане `tasks-worker-split`. (Ортогонален будущему `SCHEDULER_ENABLED` оттуда.)

**Отвергнутые альтернативы:**
- Глобальный мутабельный реестр `set_guard(label, fn)` с рантайм-подменой `disabled` (ранний черновик).
  Наш реестр — другой: он строится один раз при сборке `create_app` (модули регистрируют виды), дальше не
  мутируется; нет рантайм-подмены и глобального изменяемого состояния.
- Фиксированный набор именованных слотов в ядре (`dict["auth"|"access"|..., Depends]`) — «зафиксировать
  права в ядре», против гибкости. Реестр `вид → guard`, заполняемый модулями, оставляет ядро агностичным
  к тому, какие виды/права существуют (ядро знает лишь `public`/`disabled`).
- Передача guard'ов **цепочкой** (`Module.zone_guards: dict[zone, list[Depends]]`, колбэки без имён) —
  заменена реестром: имена дают `@guard("вид")` ссылку по имени и **валидацию наличия** (с колбэками без
  имён проверить опечатку в `@guard(...)` нельзя).

## Эскиз `core_users` + примеры маршрутов

Эскиз кода (вне области реализации — `core_users` отдельным заходом; финальные сигнатуры дошлифуем).
Показывает: метки несут непрозрачные данные, права — коды CASL, ядро словарь прав не знает, `core_users`
**регистрирует свои guard'ы в реестре ядра** при регистрации модуля, а `create_app` валидирует, что все
`@guard(...)` ссылаются на зарегистрированные виды.

### Ядро — один декоратор `@guard` (только данные, без логики прав)

```python
# src/core/router/labels.py — @guard кладёт на endpoint непрозрачное правило (вид, *аргументы)
def guard(kind: str, *args: str):
    def deco(fn):
        fn.__guards__ = (*getattr(fn, "__guards__", ()), (kind, *args))
        return fn                                       # ядро вид/аргументы НЕ парсит — просто хранит
    return deco

def guard_rules(ep) -> tuple:   return getattr(ep, "__guards__", ())          # все правила маршрута
def is_public(ep) -> bool:       return ("public",) in guard_rules(ep)        # виды public/disabled знает само ядро
def is_disabled(ep) -> bool:     return ("disabled",) in guard_rules(ep)
```

Декоратор только **хранит** `(вид, *аргументы)` — проверить наличие вида в момент декорирования нельзя
(декоратор выполняется при импорте маршрутов, а guard'ы регистрируются позже, в `create_app`). Проверку
делает проход валидации на сборке (ниже).

### Ядро — общий реестр guard'ов (плоский `вид → guard`, не зонный)

```python
# src/core/guards/registry.py — один общий реестр, не разделён по зонам
class GuardRegistry:
    def __init__(self):
        self._guards = {}                                   # вид → guard
    def add(self, kind: str, guard) -> None:
        if kind in self._guards:
            raise ValueError(f"guard '{kind}' уже зарегистрирован")
        self._guards[kind] = guard
    def has(self, kind: str) -> bool:    return kind in self._guards
    def resolve(self, kind: str):        return self._guards[kind]
```

### Этот план — мок `auth`/`ability` (временно, в `src/core/api/auth.py`)

```python
# src/core/api/auth.py — рядом с уже существующим _MOCK_USER (admin, abilities ≈ manage:all)
```

```python
# core_users/guards.py — РЕАЛЬНАЯ логика (в модуле, не в ядре)
from src.core.api import ApiError
from src.core.router import guard_rules              # читает правила с endpoint'а

async def auth_guard(request: Request) -> None:
    user = await _user_from_session_cookie(request)   # httpOnly-cookie → User
    if user is None:
        raise ApiError.unauthorized("Сессия истекла")
    request.state.user = user                         # для следующих guard'ов/хендлера

async def access_guard(request: Request) -> None:
    for kind, *args in guard_rules(request.scope["endpoint"]):   # () ⇒ no-op
        if kind == "ability":                                     # новый вид ⇒ ещё одна ветка, ядро не трогаем
            for code in args:                                     # AND: нужны все коды
                action, subject = code.split(":", 1)              # "admin:read"
                if not request.state.user.can(action, subject):   # CASL
                    raise ApiError.forbidden(f"Нет права {code}")
```

```python
# core_users/module.py — РЕГИСТРИРУЕТ guard'ы в реестре (шов = метод регистрации модуля)
class CoreUsersModule(Module):
    name = "core_users"
    router_internal = auth_router                     # /auth/login (@guard("public")), /auth/me ...
    def register_guards(self, registry: GuardRegistry) -> None:
        registry.add("auth", auth_guard)              # вид "auth" → реализация
        registry.add("ability", access_guard)         # вид "ability" → CASL
```

```python
# src/core/router/labels.py — zone_guard через реестр; валидация видов
from src.core.guards import disabled  # фолбэк

def make_zone_guard(registry, default: list[str]):    # default = умолчательные виды зоны (параметр монтажа)
    async def zone_guard(request: Request) -> None:
        ep = request.scope["endpoint"]
        if is_public(ep):    return await registry.resolve("public")(request)    # всё разрешает
        if is_disabled(ep):  return await registry.resolve("disabled")(request)  # всё кладёт
        kinds = (*default, *(k for k, *_ in guard_rules(ep)))
        for kind in (kinds or ["disabled"]):          # нет ни умолчания, ни меток ⇒ disabled
            await registry.resolve(kind)(request)     # auth (умолчание) → ability (метка), по порядку
    return zone_guard

def _validate_guard_rules(app, registry) -> None:     # вызывается на сборке, реестр уже полон
    for route in app.routes:
        ep = getattr(route, "endpoint", None)
        for kind, *_ in (guard_rules(ep) if ep else ()):
            if not registry.has(kind):
                raise RuntimeError(f'@guard("{kind}") на {route.path}: вид не зарегистрирован')
```

## Шаги

1. Скелет зон ядра — `src/core/router/`: 3 файла зон + `labels.py` (декоратор `@guard("вид",*арг)` —
   кладёт непрозрачные правила; `guard_rules`/`is_public`/`is_disabled`; `make_zone_guard(registry, zone)`
   — `public`/`disabled` исполняет сам, иначе умолчание зоны + виды меток через реестр, фолбэк `disabled`;
   `_validate_guard_rules`) + `__init__.py`. Файлы: `src/core/router/*`.
2. `src/core/guards/` — `GuardRegistry` (add/has/resolve) + два встроенных guard'а: `disabled`

   `src/core/guards/*`, `src/core/middleware/*`.
3. **Мок `auth`/`ability` (этот план)** — `mock_auth_guard` (кладёт `_MOCK_USER`) + `mock_ability_guard`

4. Контракт модуля + центральный монтаж + **шов реестра** — добавить в `Module` под-роутеры зон и метод
   `register_guards(registry)` (по умолчанию no-op); в `create_app` агрегировать под-роутеры, **собрать

## Тесты

- Unit: реестр — `add` дублем ⇒ ошибка; `has`/`resolve`. (Валидация `@guard`/default зоны — следующий шаг.)
- Unit: метки — `guard_rules` собирает правила из нескольких `@guard`; `is_public`/`is_disabled` читают
  встроенные виды.
- Unit: встроенные guard'ы — `@guard("public")` ⇒ доступ всегда (даже без умолчания); `@guard("disabled")`
  ⇒ закрыт всегда (даже при умолчании); маршрут без умолчания и меток ⇒ закрыт фолбэком `disabled` (401).
- Unit: исполнение через реестр (тест-guard'ы из фейкового модуля) — умолчание зоны → виды меток по
  порядку, первый `raise` останавливает; первый кладёт `request.state.user`, второй читает правила.
- Unit: мок `auth`/`ability` — `mock_auth_guard` кладёт `_MOCK_USER`; `mock_ability_guard` пропускает
  любой `@guard("ability",…)` (admin: read/write/all).
- Integration (с моком): `GET /internal/<m>/...` ⇒ 200 (мок auth); `@guard("ability","admin:read")` ⇒ 200
  (мок admin); `@guard("public")` — всегда.
- Unit: `BACKEND_API_ENABLED` — `false` (дефолт) ⇒ зон-роутеры не смонтированы (любой `/internal/...` ⇒
  404), lifespan/scheduler жив; `true` ⇒ зоны смонтированы; CLI перекрывает env.
- Регресс: существующие API-тесты модулей зелёные под префиксами `/internal/<m>`; `pytest --core` зелёный.
  ВАЖНО: дефолт `BACKEND_API_ENABLED=false` ⇒ тест-фикстуры, поднимающие API, должны включать флаг
  (`true`), иначе зоны не смонтированы и все API-тесты ⇒ 404.

## Проверка

- С мок-guard'ами этого плана (`BACKEND_API_ENABLED=true`): защищённый `GET /internal/<m>/...` ⇒ 200
  (мок `auth`); `@guard("ability","admin:read")` ⇒ 200 (мок admin); `@guard("public")` — открыт.
- `BACKEND_API_ENABLED=false` (дефолт): `GET /internal/...` ⇒ 404 (не смонтировано), HTTP-маршрутов нет;
  scheduler/выполнение задач работают.
- `api`/`webhook` в этом плане не смонтированы (отключены).
- Фронт-smoke: список грузится через обёртку против `/internal/...`.

## Решённые развилки

- **Механика «public» — метка-opt-out (A), реализованная guard'ом.** Default-on на зону +
  `@guard("public")` снимает умолчание (модель «вешаем на все, снимаем для login»). Отвергнут вариант B
  (отдельный public-роутер на зону) — публичных маршрутов мало, метка читаемее и держит зону единой в
  OpenAPI. `public` — не спец-случай, а универсальный guard «всё разрешает» в `src/core/guards/`; парный
  ему `disabled` «всё кладёт» (он же фолбэк без умолчания/меток). Эти два вида ядро знает и исполняет само.
- **Один явный декоратор `@guard("вид", *аргументы)`** вместо пары `@public`/`@requires`. Кладёт на
  endpoint непрозрачное правило `(вид, *аргументы)`; ядро его не парсит (кроме опт-аута `public`),
  виды разбирают guard'ы из `core_users`. Имя `@guard` выбрано пользователем (раньше рассматривался
  `@access`); отвергнут `@middleware` (механически это НЕ ASGI-middleware, а метаданные, читаемые
  зависимостью — путало бы + коллизия с папкой `src/core/middleware/`). Слово «guard» в коде сознательно
  значит и метку-объявление (`@guard`), и проверяющую зависимость (`auth_guard`/`access_guard`,
  `src/core/guards/`): **метка указывает на исполнителя** — одна логика, один модуль. Декоратор только
  навешивает данные; саму проверку делает зависимость.
- **Проводка guard'ов — общий (не зонный) реестр `вид → guard`, заполняемый при регистрации модуля.**
  `core_users` в `register_guards(registry)` делает `add("auth",…)`/`add("ability",…)`; ядро
  предрегистрирует `public`/`disabled`; default-on зоны — параметр монтажа (`make_zone_guard(registry,
  default=["auth"])`), не свойство реестра. `create_app` строит реестр один раз и **валидирует** все
  `@guard(...)` (незарегистрированный вид ⇒ ошибка на старте — «guard зарегистрирован до использования»;
  проверка на сборке, т.к. декоратор отрабатывает при импорте, раньше). Отвергнуты: глобальный мутабельный
  реестр с рантайм-подменой; фиксированные слоты в ядре; передача безымянной цепочкой (`zone_guards`) —
  без имён нельзя валидировать `@guard("вид")`. Не прямой импорт `core_users` ядром.
- **Форма прав — CASL-коды через `@guard("ability", *codes)`.** Коды `action:subject` (`admin:read`) —
  непрозрачные строки; ядро их не парсит. `access_guard` из `core_users` (вид `ability` в реестре) читает
  правила вида `ability` (`guard_rules`) и сверяет с `user.abilities` по CASL. Перечень прав в ядре НЕ
  фиксируется (гибкость); отвергнут фиксированный enum уровней/ролей. Несколько кодов — **AND**; OR — позже.
