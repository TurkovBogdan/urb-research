## i18n — multi-language support

> **Status: draft, in progress.** Conventions below are the agreed starting point; expect refinements as we add real translated screens.

Supported locales: `ru` (**authoring** primary), `en`. Russian is the source of truth for *authoring*: new strings are written in `ru` first.

**Fallback is `en` by design** (`fallbackLocale: 'en'` in `web/src/plugins/i18n.ts`). "Primary" here means the *authoring* language, not the fallback target: when the active locale lacks a key, vue-i18n renders the **English** value. Since `ru` keys are authored first they are rarely the ones missing; `en` is the deliberate safety net for partially-translated screens (the i18n migration is incremental — most screens are still ru-only). Do **not** "fix" this to `fallbackLocale: 'ru'` — `en` is the intended direction.

## Frontend — `vue-i18n`

### Library

- `vue-i18n@^11` registered in `web/src/plugins/i18n.ts` and wired in `main.ts` before `vuetify`.
- Vuetify locale is kept in sync via `createVueI18nAdapter` (`plugins/vuetify.ts` → `locale.adapter`). Vuetify's own strings live under the `$vuetify` key of each locale in the i18n tree (`plugins/i18n.ts` merges `vuetify/locale`'s `ru`/`en`), so `VDataTable`, `VPagination`, date pickers, etc. follow the active language automatically — provided the component uses Vuetify defaults and not app-supplied props (several list footers are hand-built with hardcoded labels and won't switch until migrated). The adapter wants a loosely-typed `I18n<any>`, so our literal-typed instance is passed as `i18n as never` (structurally compatible at runtime).

### File layout

| Scope | Path |
|---|---|
| Common (layout, sidebar groups, generic actions/statuses, common errors) | `web/src/locales/{ru,en}.json` |
| Per-feature | `web/src/features/<feature>/locales/{ru,en}.json` |

Per-feature dictionaries are loaded via `import.meta.glob` and merged into the i18n messages tree under a namespace equal to the feature directory name (e.g. `settings`, `design-system`). The frontend feature directory should match the backend module name. A hyphen in the directory name (`design-system`) is fine — vue-i18n treats `-` as a literal character inside a path segment, so `t('design-system.page.buttons.title')` resolves normally.

**Root-lifted namespace (NOT glob-derived).** `design-system` lives at `web/src/locales/design-system/{ru,en}.json` and is **imported explicitly** in `i18n.ts`, merged into the `ru`/`en` trees by hand. It was dissolved as a feature (template chrome, not a module): views moved to `@/views/design-system/`, routes to `@/router/design-system.ts`, nav inlined in `AppSidebar` — but the namespace + URLs are unchanged.

**Array values** (e.g. lists of tag chips) are read with `tm(key)` and each item rendered through `rt(item)` — see `DesignSystemIndexView.vue` (`design-system.index.page.<slug>.tags`).

### Key naming

- **Plain English snake_case identifiers.** Keys describe the role, not the text.
  - `settings.nav.settings` — name of the "Settings" link in the sidebar
  - `settings.filter.count.label`, `settings.filter.count.hint`
- **Group by UI location first, then by element.** Recommended top-level buckets inside a feature namespace:
  - `nav` — sidebar/menu labels
  - `page` — page-level titles, subtitles
  - `form.<field>.label` / `form.<field>.placeholder` / `form.<field>.hint`
  - `table.header.*`
  - `action.*` — buttons, menu items
  - `status.*` — chips, badges
  - `error.*` — error messages (see below)
- **Do not group by element type at top level** (`labels.*`, `buttons.*` — does not scale).
- **Do not translate** machine codes: enum values, task statuses in API payloads, route paths, technical module names, proper nouns (third-party product/brand names).

### Parameter interpolation

`vue-i18n` `{name}` syntax:

```json
{ "error": { "email": { "value_taken": "Email {value} уже занят" } } }
```

```ts
t('<feature>.error.email.value_taken', { value: 'user@example.com' })
```

### Fallback policy

- `fallbackLocale: 'en'` — by design (see the top of this file): a key missing in the active locale renders its English value.
- Missing key entirely (no value in the active locale **and** no fallback value) renders the **key itself**, not an empty string — makes gaps obvious in the UI and greppable.

### Language switching

- `setLocale(locale)` in `plugins/i18n.ts` is the single entry point: sets the active vue-i18n locale, persists to `localStorage` (`app.locale`), and updates `document.documentElement.lang`. Vuetify follows via the adapter. Never set `i18n.global.locale` directly — go through `setLocale`.
- The language switcher lives on the **profile** page, in the «Язык и регион» card (`web/src/features/user-profile/views/ProfileView.vue`), alongside timezone and date-format prefs. (It was removed from the sidebar and from the Settings page — see tasks `2026-06-06-settings-remove-language` / `2026-06-06-frontend-datetime-prefs`.) Option labels are endonyms (`Русский` / `English`) and are intentionally not translated.
- Selection persists in `localStorage` (single local installation; per-user DB column deferred).

## Backend — error codes, not translated strings

Reports and other long-form text from the backend are returned as-is (no translation layer planned). For **errors**, the backend returns machine codes; the frontend translates.

### Backend-owned display strings → derive the key from the stable identity

Some backend objects carry human-facing `name`/`description` literals (e.g. settings `Field` metadata per module schema). These are **not** translated on the backend and the backend does **not** send a translation key. Instead the frontend derives the key from the object's stable machine identity that's already in the payload:

```
field identity:  (module, code)  e.g. ("my_module", "session_idle")
i18n key:        <module>.field.<code>.{label,hint}
```

Rules (see `features/settings/settingText.ts` — `localizeField`):
- **Derive, don't transmit.** `(module, code)` *is* the key — adding a `*_key` field to every backend object would be redundant and couple the layers.
- **Always fall back to the backend literal.** Lookup order: the module's own namespace → a catch-all bucket for owners without a frontend namespace (`settings.catalog.<module>.<code>`; `core` has no feature dir) → the backend literal. So a newly-added or untranslated field renders its (Russian) backend literal, never a raw key. Use `te()` to test existence before `t()`.

### Response shape

```json
{ "error": { "code": "my_module.user.not_found", "params": { "id": 42 } } }
```

- `code` — `<module>.<entity>.<reason>`, snake_case. HTTP status remains meaningful (404/409/422) and is not duplicated in `code`.
- `params` — interpolation data only, never pre-rendered text.

### Code registry per module

Codes live as constants on the module side to avoid string-literal sprawl and to give a greppable inventory:

```python
# src/modules/my_module/errors.py  (illustrative — the raising mechanism is still TBD, see below)
class MyModuleErrors:
    USER_NOT_FOUND = "my_module.user.not_found"
    EMAIL_VALUE_TAKEN = "my_module.user.email_taken"
```

> Raising mechanism (custom `AppError` + FastAPI exception handler) — **TBD**, will be designed when we convert the first module.

### Frontend resolution

Module-scoped errors are looked up inside the matching feature dictionary, under an `error` bucket whose path mirrors the code:

```
backend code:  my_module.user.not_found
i18n key:      my_module.error.user.not_found
```

A single helper inserts the `error` segment:

```ts
function tError(err: { code: string; params?: Record<string, unknown> }) {
  const [feature, ...rest] = err.code.split('.')
  return t(`${feature}.error.${rest.join('.')}`, err.params ?? {})
}
```

### Common (non-module) errors

Generic, non-feature error strings (e.g. the 404 page) live in the shared `common.errors.*`
bucket of `web/src/locales/{ru,en}.json` (`common.errors.notFound.*`, `common.errors.action.*`).

## Frontend store & loading patterns (set by the migrated features)

These conventions emerged alongside the i18n migration and are the reference for new translated screens:

- **Pinia store per page/concern**, file `<name>.store.ts` (each `items`/`loading`/`error`/`load(force)`). There is **no persist plugin** in the repo → persist manually via `localStorage` (like `setLocale`).
- **Stale-while-revalidate index.** An index store can cache `counts` under a `localStorage` key and hydrate on init so revisits render instantly (skeleton shown only when `counts === null`, i.e. first-ever visit); `refresh()` reloads the entity stores in the background and updates counts in place — no gray-plate jump.
- **Skeleton loading standard** (showcase `views/design-system/feedback/SkeletonView.vue`; design-system views are grouped by purpose into `basics`/`controls`/`data`/`charts`/`feedback`/`content`/`structure/`, index `DesignSystemIndexView.vue` at the dir root): wrap with `VSkeletonLoader :loading type=...`; for tables use `type="table-heading, table-row-divider@N"` shown while `loading && list.length === 0`, then render the real `VDataTable`; small counts use `type="text"`.

## What is intentionally **not** done yet

- `AppError` + FastAPI exception handler — backend still raises `HTTPException(detail="…")`.
- Bulk migration of existing Russian literals — done incrementally per feature, not in a sweep. Most strings (page titles, form labels, list-footer labels, status chips) are still hardcoded Russian. Migrated so far: `design-system` (showcase; buckets `nav`/`index`/`page`/`section`); and `settings` (Settings; reference pattern — chrome + the backend-owned-identity pattern above for `Field` metadata via `settingText.ts` `localizeField`, applied through a reactive `localizedModules` computed in the view so the 8 `SettingFieldXxx` leaf components stay untouched and labels switch live; `settings.catalog.*` catch-all). Genuinely shared (non-feature) component strings go to the `common` namespace (`web/src/locales`) — e.g. `common.action.add` for `SettingFieldList`. The design-system migration set a **partial-translation convention** worth reusing for showcase/demo screens: translate the catalog + page title/description + categorical section headings, but keep demo/example content as static English (no i18n), and keep identifiers/component/prop/CSS names literal.

`globalInjection: true` is set on `createI18n`, so `$t`/`$te` are available in templates — use them where a `v-for` loop variable shadows the `t` from `useI18n` (e.g. `v-for="t in tasks"`).
- Per-user persistence of the locale choice.

These will be added as separate tasks; until then the rules above are the standard for any new translated string.
