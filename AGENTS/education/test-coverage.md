# Покрытие тестами (coverage): как мерить и как читать

Урок про инструмент `pytest-cov` (поверх `coverage.py`): что он показывает, как им
пользоваться **в этом проекте**, и — главное — почему «процент покрытия» легко
обманывает. Примеры из реальной работы над токенами `core_users`.

---

## 1. Что такое coverage и зачем он

**Coverage (покрытие) — это измерение того, какие строки твоего кода реально
исполнились, пока шли тесты.** Инструмент запускает тесты с включённым «трейсером»:
на каждую исполненную строку ставит галочку. В конце показывает, что осталось без
галочки — то есть код, который **ни один тест ни разу не выполнил**.

Зачем это нужно: тесты могут «проходить» и при этом обходить стороной целые ветки —
обработку ошибок, редкие `if`, ранние `return`. Такой код формально «работает», но на
деле **никем не проверен**. Coverage подсвечивает именно эти слепые зоны.

Аналогия: ты проверяешь новую квартиру и щёлкаешь каждым выключателем. Coverage — это
список выключателей, которые ты **не** тронул. Он не говорит, что свет горит правильно
(это работа теста-ассерта), он говорит лишь «вот сюда ты вообще не заходил».

Это сразу задаёт границу применимости (важно — см. §6): **coverage отвечает на вопрос
«исполнилось ли?», а не «проверено ли правильно?»**. Сто процентов покрытия с пустыми
ассертами — бесполезны.

---

## 2. Инструмент в этом проекте

Тесты гоняются через `uv run pytest` (venv не на PATH — всегда префикс `uv run`, см.
`AGENTS/docs/workflow/testing.md`). Для покрытия добавлен `pytest-cov` в группу зависимостей
`test` (в `pyproject.toml`, ставится вместе с остальным через `uv sync --all-groups`).

Базовый запуск — замерить покрытие пакета:

```bash
# -n0 обязателен (см. §5); БД — in-memory sqlite, отдельной настройки не нужно
uv run pytest --module=core_users -n0 \
  --cov=src/modules/core_users \
  --cov-report=term-missing
```

Ключевые флаги:

- `--cov=<путь>` — что измерять. Можно пакет (`src/modules/core_users`) или один файл
  (`src/modules/core_users/service/token.py`). Несколько `--cov=` складываются.
- `--cov-report=term-missing` — печатать таблицу в терминал **со списком непокрытых
  строк** (`Missing`). Без `term-missing` будет только процент.
- `--cov-report=html` — сгенерировать кликабельный HTML-отчёт в `htmlcov/` (открыть
  `htmlcov/index.html`): подсвечивает каждую строку зелёным/красным. Удобно для глаз.

## 3. Как читать таблицу

Пример вывода из работы над токенами:

```
Name                                          Stmts   Miss  Cover   Missing
-----------------------------------------------------------------------------
src/modules/core_users/security.py               33      0   100%
src/modules/core_users/crud/token.py             45      0   100%
src/modules/core_users/service/token.py          64      1    98%   90
src/modules/core_users/api.py                    88     10    89%   89-90, 158, 174, 189
```

Колонки:

- **Stmts** — сколько исполняемых инструкций в файле (не считая пустых строк/комментов).
- **Miss** — сколько из них **не** исполнилось ни в одном тесте.
- **Cover** — процент = `(Stmts − Miss) / Stmts`.
- **Missing** — номера непокрытых строк. **Это самое полезное поле** — идёшь и смотришь,
  что там за код.

Процент сам по себе ничего не значит. Работаешь именно со столбцом `Missing`: открываешь
каждую строку и решаешь — это реальная дыра (нужен тест) или нет.

## 4. Не всякая непокрытая строка — дыра

Разбор реальных остатков из таблицы выше — три разных вердикта:

### (а) Реальная дыра → пишем тест

`service/token.py:62` была непокрыта:

```python
if len(name) > _MAX_NAME:
    raise ApiError.validation(fields={"name": f"Максимум {_MAX_NAME} символов"})
```

Тест на пустое имя был, а на **слишком длинное** — нет. Это валидация ввода на боевом
эндпоинте → реальная дыра. Добавили `test_create_token_name_too_long_422`. Аналогично
вскрылась необработанная ветка **коллизии `public_id`** (ретрай генератора) — добавили
два теста через monkeypatch `security.new_token`.

Это и есть главная польза coverage: он **нашёл забытые ветки**, о которых легко не
вспомнить, перечисляя тест-кейсы по памяти.

### (б) Недостижимый защитный код → оставляем как есть

`service/token.py:90` так и осталась непокрытой — и это нормально:

```python
for attempt in range(_PUBLIC_ID_ATTEMPTS):
    ...
    return full_token, row          # всегда выходим отсюда
raise AssertionError("unreachable")  # ← строка 90: сюда попасть нельзя
```

Цикл всегда либо возвращает значение, либо поднимает `conflict` внутри. Финальный
`assert` — страховка «если кто-то сломает логику цикла». **Его невозможно покрыть
тестом, не сломав инвариант.** Гнаться за 100% тут — вредно. Правильный ход: осознать,
что строка недостижима, и оставить.

### (в) Артефакт инструмента → не дыра, проверяем иначе

`api.py:158` (`return CreateTokenResponse(...)`) числится непокрытой, **хотя тест
`test_manager_create_token_returns_raw_once` проверяет тело этого ответа** — значит
строка точно исполнилась. Почему «красная»? Трейсер `coverage.py` в некоторых версиях
не фиксирует завершающие строки `async def`-эндпоинтов, которые FastAPI вызывает через
свой исполнитель. Это **известный артефакт**, а не пробел.

Как отличить артефакт от дыры: посмотри, есть ли тест, который **проверяет результат**
этой строки. Если ассерт на её эффект проходит — строка исполняется, цифра врёт. Если
такого ассерта нет — это настоящая дыра.

**Вывод раздела:** coverage даёт *список кандидатов*, а вердикт по каждому выносишь ты.
Три исхода: написать тест / признать недостижимым / распознать артефакт.

## 5. Подвох именно у нас: xdist + cov не дружат

Проект по умолчанию гоняет тесты параллельно (`pytest-xdist`, `-n auto`; у каждого воркера своя in-memory sqlite).
В текущих версиях (`pytest 9` + `xdist 3.8` + `pytest-cov`) запуск coverage **под
параллелью падает** с `AttributeError: 'Central' object has no attribute
'configure_node'` ещё на старте сессии.

Поэтому coverage у нас меряется **в один процесс**: флаг `-n0`. Это не костыль, а
нормальная практика — сбор покрытия по нескольким воркерам и так требует отдельной
склейки `.coverage`-файлов и капризен. Один процесс = простой и точный замер.

Следствие: замер покрытия идёт медленнее обычного прогона (нет параллели). Это
терпимо — coverage запускают точечно («проверить новую фичу»), а не на каждый чих.
Обычный прогон тестов оставляй параллельным (без `--cov`).

Полезные напарники к `-n0`:
- `--module=<name>` / `--core` — сузить набор тестов под измеряемый код.

## 6. Чего coverage НЕ умеет (чтобы не обманываться)

1. **Не проверяет правильность.** Строка исполнилась ≠ поведение верное. Это работа
   ассертов. Тест без `assert`, но «прошедший по строкам», даёт 100% и нулевую ценность.
2. **Line coverage ≠ branch coverage.** По умолчанию мерятся *строки*. Строка
   `x = a if cond else b` засчитается покрытой, даже если `cond` всегда был `True` —
   вторая ветка не проверена, а строка «зелёная». (Включается режим веток
   `--cov-branch`, но и он не панацея.)
3. **Не видит недостающих кейсов данных.** Если функция падает на пустом списке, а тест
   всегда передаёт непустой — обе строки покрыты, баг живёт. Coverage про *код*, не про
   *входные данные*.

Поэтому правильный порядок мыслей: **сначала продумай кейсы по смыслу** (нормальный путь,
границы, ошибки, права доступа), напиши тесты с ассертами — **а coverage запусти в конце**,
чтобы поймать *забытые* ветки. Не наоборот («гоним процент вверх любой ценой»).

Так и делали с токенами: сначала тесты на cross-scope / revoked / expired / tampered /
wrong-pepper / чужой токен → 403, а coverage потом подсветил ещё пару пропущенных веток
валидации. Он — финальная страховочная сетка, а не основа стратегии.

## 7. Чеклист «померить покрытие фичи»

1. Запустить: `uv run pytest --module=<m> -n0 --cov=src/modules/<m> --cov-report=term-missing`.
2. Смотреть **столбец `Missing`**, не процент.
3. По каждой непокрытой строке вынести вердикт:
   - реальная ветка (валидация/ошибка/редкий if) → **написать тест с ассертом**;
   - недостижимый защитный код (`assert "unreachable"`) → **оставить**;
   - завершающий `return` async-эндпоинта при наличии ассерта на ответ → **артефакт, не дыра**.
4. Дописать тесты на реальные дыры, перезапустить — убедиться, что `Missing` сократился
   до объяснимого остатка.
5. Не путать «100%» с «протестировано»: проверь, что у ключевых веток есть **ассерты на
   поведение**, а не просто заход в строку.

---

## Короткое резюме

- `pytest-cov` показывает, какие строки **исполнились** под тестами; работаешь со столбцом
  `Missing`, а не с процентом.
- Запуск у нас: `uv run pytest ... -n0 --cov=... --cov-report=term-missing` (**`-n0`
  обязателен** — параллель xdist с cov сейчас падает).
- Непокрытая строка — это *кандидат*, не приговор: тест / недостижимый код / артефакт async.
- Coverage отвечает «исполнилось ли», НЕ «правильно ли» — сначала продумай кейсы и ассерты,
  coverage запусти в конце как страховку от забытых веток.
