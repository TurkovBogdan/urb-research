# Ленивая загрузка (lazy imports) в Python

Урок про то, как работает «ленивая загрузка» модулей и почему она уменьшает память
процесса. С примерами прямо из нашего кода (`llm_providers`, `core_nlp`).

---

## 1. Сначала — что вообще делает `import`

Когда Python выполняет строку

```python
import anthropic
```

происходит не «объявление», а **реальная работа прямо в момент выполнения этой строки**:

1. Python находит пакет `anthropic` на диске.
2. Берёт его `__init__.py` и **исполняет его сверху вниз** — как обычный скрипт.
3. Тот, в свою очередь, импортирует свои подмодули, они — свои, и так далее по цепочке.
4. Все получившиеся объекты (классы, функции, словари, скомпилированные регулярки)
   остаются жить в памяти процесса — в кэше `sys.modules`.

Ключевая мысль: **импорт — это не бесплатная декларация, а исполнение кода и выделение
памяти.** Большая библиотека на импорте может «съесть» десятки мегабайт ещё до того, как
ты вызвал хоть одну её функцию.

Проверить можно прямо в нашем проекте — мы это и замеряли:

```
anthropic   +10 MB
openai      +10 MB
mcp         +11 MB
```

Это RSS (реально занятая процессом память), которая прибавилась **только от факта импорта**,
без единого вызова.

### Где именно происходит импорт

Импорты бывают на **верхнем уровне** файла (module-level) и **внутри функции** (local).

```python
# верхний уровень — исполняется СРАЗУ, как только кто-то импортирует этот файл
import anthropic

def call_claude():
    ...
```

```python
def call_claude():
    # локальный — исполняется ТОЛЬКО когда вызвали эту функцию
    import anthropic
    ...
```

Разница в том, **когда** Python заплатит за импорт памятью. В этом и весь фокус.

---

## 2. Что такое «ленивая загрузка»

**Ленивая загрузка = откладывание импорта тяжёлой библиотеки до момента, когда она
действительно понадобилась.** Технически это чаще всего просто перенос строки `import`
с верхнего уровня файла внутрь функции/метода.

Аналогия: не тащить с собой весь набор инструментов «на всякий случай» в начале рабочего
дня, а доставать дрель из шкафа только когда реально нужно сверлить. Если за весь день
сверлить не пришлось — дрель так и не покинула шкаф, руки свободны.

«Лениво» здесь — обычный программистский термин: *вычислять/загружать по требованию, а не
заранее.* Тот же смысл, что в «ленивых» итераторах или `@property`, которое считается при
первом обращении.

## 3. Почему это важно именно у нас

Напомню топологию (см. задачу `2026-06-05-memory-footprint-reduction`):

- В проде запускается **несколько web-процессов** (`SERVER_PROCESSES`) + отдельный **worker**.
- **Каждый** процесс при старте вызывает `build_modules()` → импортирует **все 11 модулей**
  со всеми их библиотеками.

Проблема: **web-процесс никогда не вызывает LLM** (это делает агент в worker'е), но из-за
жадных импортов всё равно тащит в память `anthropic` + `openai` + `mcp` = **~31 MB зря**.
И так на каждом web-процессе.

Цепочка, по которой это происходит сейчас:

```
src/modules/llm_providers/__init__.py
  └─ module.py            (import client, tools.mcp.manager)
       └─ client.py       → providers/__init__.py
                              ├─ providers/anthropic.py   → import anthropic   (верхний уровень!)
                              └─ providers/openai_compat.py → import openai     (верхний уровень!)
       └─ tools/mcp/manager.py → tools/mcp/client.py → import mcp               (верхний уровень!)
```

Достаточно, чтобы `llm_providers` просто **зарегистрировался как модуль** — и три тяжёлые
SDK уже в памяти, хотя их код может не выполниться ни разу за всю жизнь web-процесса.

## 4. Как сделать ленивым — на нашем примере

### Было (жадно)

`src/modules/llm_providers/providers/anthropic.py`:

```python
import anthropic   # ← верхний уровень: грузится при импорте модуля

class AnthropicProvider(LlmProvider):
    def __init__(self, api_key: str):
        self._client = anthropic.AsyncAnthropic(api_key=api_key)
```

### Стало (лениво)

```python
from typing import TYPE_CHECKING

if TYPE_CHECKING:
    import anthropic   # видно только статическим анализаторам (mypy/IDE), в рантайме НЕ исполняется


class AnthropicProvider(LlmProvider):
    def __init__(self, api_key: str):
        import anthropic            # ← платим за импорт ТОЛЬКО при создании провайдера
        self._client = anthropic.AsyncAnthropic(api_key=api_key)
```

Что изменилось по шагам:

1. **`import anthropic` с верхнего уровня убрали** — теперь импорт файла
   `anthropic.py` не тянет SDK.
2. **`import anthropic` внутри `__init__`** — SDK подгрузится в тот момент, когда кто-то
   реально создаёт `AnthropicProvider(...)`. В web-процессе этого не происходит → 10 MB
   сэкономлены. В worker'е происходит при первом запуске агента → те же 10 MB, просто чуть
   позже (один раз на процесс, дальше из кэша `sys.modules` — мгновенно).
3. **Блок `if TYPE_CHECKING:`** — это спецконстанта: в обычном запуске она `False`, поэтому
   строка не исполняется и память не тратит. Но mypy, IDE и автодополнение считают её `True`
   и видят тип `anthropic.AsyncAnthropic`. То есть подсказки типов сохраняются, а рантайм-цена
   — нет. Нужен только если в аннотациях фигурируют типы из этой библиотеки.

### Важный нюанс: импорт кэшируется

Локальный `import` **не** перезагружает библиотеку при каждом вызове функции. Первый вызов
исполняет `__init__.py` и кладёт результат в `sys.modules`; все последующие `import`
(хоть в цикле, хоть в горячем пути) — это просто поиск готового объекта в словаре,
**наносекунды**. Поэтому «лениво» не значит «медленно при каждом вызове» — цена платится
ровно один раз на процесс.

## 5. Где ленивость НЕ нужна (чтобы не переусердствовать)

Ленивый импорт — это лёгкий минус к читаемости (import не на привычном месте, вверху файла).
Поэтому делаем это **только для тяжёлых и редко используемых** библиотек. Примерная граница:

- **Делать лениво:** большие сторонние SDK, которые нужны не всем процессам —
  `anthropic`, `openai`, `mcp`, тяжёлые парсеры. Выигрыш в мегабайтах.
- **Не трогать:** стандартная библиотека (`os`, `json`, `asyncio`), мелочь, и всё, что и так
  используется почти в каждом запросе. Экономия ноль, а читаемость хуже.

Правило: **ленивый импорт оправдан, если библиотека (а) весит заметно и (б) нужна не на
каждом пути исполнения.** `anthropic`/`openai`/`mcp` проходят оба условия идеально.

## 6. Отдельный случай — «ленивый синглтон» (и анти-паттерн eager-warmup)

Иногда дорогое — не сам импорт, а **инициализация данных** (загрузка словаря/модели в объект).
Тут помогает родственный приём — **ленивый module-level singleton**: объект создаётся при
первом обращении, а не при импорте файла:

```python
_analyzer = None

def _get_analyzer():
    global _analyzer
    if _analyzer is None:                 # создаём один раз на процесс
        _analyzer = HeavyThing()          # дорогая инициализация — по требованию
    return _analyzer
```

Это та же идея «по требованию», но на уровне **объекта-данных**, а не импорта.

**Анти-паттерн, который мы реально словили (и убрали 2026-06-05):** у модуля `core_nlp`
была `morph.py` с таким ленивым синглтоном поверх словаря pymorphy3 — НО `configure()`
модуля принудительно звал `warmup()`, который строил анализатор у **каждого** процесса на
старте. То есть ленивость «съедалась» вручную: словарь грузился в `create_app` всем
процессам (~19 MB RSS), хотя функции морфологии (`lemma`/`to_nominative`) не вызывались
**нигде** в коде. Итог: и сам код, и зависимость `pymorphy3` удалены как мёртвые — пол
процесса упал 158 → 139 MB.

Два урока:
1. Ленивый синглтон не помогает, если где-то есть **eager-warmup**, дёргающий его на старте.
   Не грузи словари/модели в `configure()` — он выполняется у всех процессов, включая web.
2. Прежде чем «оптимизировать» загрузку чего-то — проверь, **вызывается ли оно вообще**
   (`grep` по вызовам). Мёртвый код выгоднее удалить, чем делать ленивым.

## 7. Как проверить, что сработало

Память процесса меряем по RSS. Простой замер «до/после» (Linux):

```python
import resource
def rss_mb():
    return resource.getrusage(resource.RUSAGE_SELF).ru_maxrss // 1024  # МБ

print(rss_mb())
import anthropic           # или import нашего модуля
print(rss_mb())            # разница = цена импорта
```

А факт того, что библиотека НЕ загрузилась, проверяется наличием её в `sys.modules`:

```python
import sys
from src.apps.app.modules import build_modules
build_modules()
print("anthropic" in sys.modules)   # после ленивой правки должно стать False
```

Именно так мы и замеряли исходную картину — см. цифры в задаче.

## 8. Чеклист «сделать импорт ленивым»

1. Найти `import <heavy>` на верхнем уровне файла.
2. Убедиться, что библиотека нужна **не всем** путям/процессам (иначе смысла нет).
3. Перенести `import <heavy>` внутрь функции/метода, где он реально используется.
4. Если тип из библиотеки нужен в аннотациях — добавить `if TYPE_CHECKING: import <heavy>`.
5. Проверить, что нет другого файла, который импортирует то же самое жадно (иначе SDK всё
   равно загрузится по другой цепочке).
6. Замерить: `"<heavy>" in sys.modules` после `build_modules()` → должно быть `False`;
   RSS процесса упал.

## Короткое резюме

- `import` — это исполнение кода и трата памяти **в момент импорта**, а не безобидная декларация.
- **Ленивая загрузка** = перенести `import` тяжёлой либы с верха файла внутрь функции, чтобы
  платить памятью только при реальном использовании.
- У нас это даёт ~31 MB на каждый web-процесс (anthropic/openai/mcp), которые web никогда не
  вызывает.
- Цена платится один раз на процесс (дальше — кэш `sys.modules`), типы сохраняются через
  `TYPE_CHECKING`.
- Родственный приём для дорогих **данных** (а не импортов) — ленивый синглтон, как в
  `core_nlp/morph.py`.
