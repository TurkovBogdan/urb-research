## Task

> Перевести раздел tasks. Описания задач хранятся на беке — нужно решить, как фронт получает перевод.

**Architecture decision (confirmed with user):** the backend does NOT send a translation key. Each task already has a stable machine identity `(module, code)` in the `/api/core/tasks` payload, so the frontend **derives** the key from it — same spirit as the error-code policy in `i18n.md` (backend = codes, frontend = translates). The backend `NAME`/`DESCRIPTION` stay as the canonical **fallback** so untranslated/new tasks keep rendering (never a raw key).

**Where translations live (confirmed):** per **module namespace** — `<module>.task.<code>.{name,description}` in each module's own feature dictionary.

## What was done

- **`useTaskText()`** composable (`features/tasks/taskText.ts`) — `taskName(info)` / `taskDescription(info)` with lookup order:
  1. `<module>.task.<code>.<field>` (module's own dict)
  2. `tasks.catalog.<module>.<code>.<field>` (catch-all for modules **without** a frontend feature namespace — currently `core`/heartbeat and `llm_providers`/agent_dispatch)
  3. backend literal (`info.name`/`info.description`) — final fallback.
  Uses `te()` to test existence before `t()`.
- **Per-module task dictionaries** (`task.<code>.{name,description}`, ru+en): created `locales/{ru,en}.json` in `twenty` (6), `intercom` (6), `mail_sync` (4), `conversations` (1); added a `task` bucket to the existing `core_geo` dict (1). New top-level feature namespaces now exist with just the `task` bucket — fine, future migration of those features adds more buckets.
- **`tasks` feature dict** (`features/tasks/locales/{ru,en}.json`) — UI chrome (`nav`/`page`/`updated`/`kpi`/`filter`/`badge`/`stats`/`action`/`empty`/`runs`) + the `catalog.core.heartbeat` and `catalog.llm_providers.agent_dispatch` catch-all entries. `nav.ts` → `labelKey: 'tasks.nav'`.
- **Both views wired:** `TasksView.vue` (KPIs, filter toggle/search, badges enabled/disabled/manual/TTL, 24h stats, run/logs buttons, relative-"updated" label with `{n}` interpolation, empty states) and `TaskRunsView.vue` (header via `taskName`/`taskDescription`, status filter, reactive `headers`/`STATUS_OPTIONS` computeds, no-data, logs, pagination meta with `{from}/{to}/{total}`). Status/level **enum** values (`running`/`success`/`error`/`debug`/…), cron strings, and `Payload` kept literal. `fmtNum` now locale-aware (`en-US`/`ru-RU`).
- **`globalInjection: true`** added to `createI18n` so templates can call `$t` where a `v-for="t in g.tasks"` loop variable shadows the `t` from `useI18n` (task cards use `$t(...)` for chrome, `taskName(t)`/`taskDescription(t)` for the item).

## Result

tasks feature fully bilingual. `vue-tsc --noEmit` clean. Verified in-browser ru↔en: all 20 task titles translate (incl. the catch-all `core`/`llm_providers`), KPIs/stats/badges/buttons switch, runs page headers + pagination translate. **The fallback proved itself live:** the backend had hot-split mail_sync threads into `full_build_threads` (new) + renamed `build_threads` after the first snapshot — the new task rendered its Russian backend literal instead of a raw key until I synced both `mail_sync` locale entries. Out of scope (other features): sidebar nav labels + section headings still Russian; backend task-log `message` strings stay as-is (operational/dynamic). Memory: [[i18n_frontend_setup]]. Doc: `AGENTS/docs/i18n.md`.
