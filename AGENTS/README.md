# Агентские инструкции

Слой инструкций для AI-агента: один основной файл плюс папки с индексами, которые агент читает в начале сессии.

## Состав

- `CLAUDE.md` (корень репозитория) — основной файл агента: факты о проекте, структура, правила работы, ссылки на все папки ниже.
- `agent-primary.md` — промпт/роль основного агента.
- `memory/` — непроизводные факты о проекте; индекс `MEMORY.md`; симлинк на собственную папку памяти Claude.
- `tasks/` — журнал работ, по файлу на область; индекс `INDEX.md`; запись открывается в начале запроса, закрывается по завершении.
- `docs/` — документы «почему и как устроено» (пайплайны, контракты, runbook'и); индекс `INDEX.md`.
- `plans/` — планы реализации; индекс `INDEX.md` + шаблон; создаются только по запросу.
- `research/` — находки по внешним сервисам и API; индекс `INDEX.md`; читать перед любой исследовательской задачей.
- `tools/` — shell-скрипты для повторяющихся операций; индекс `INDEX.md`.
- `reference/` — внешние справочные материалы.

У каждой папки свой индекс, который читается первым; детали лежат в папке, а не в индексе.

## В чём разница между разделами

Разделы делятся не по «типу контента» (память и docs оба хранят «то, чего нет в коде»), а по **модели загрузки** и **роли**. Одна тема живёт ровно в одном разделе.

| Раздел | Когда грузится | Что хранит | Ключевое отличие |
|---|---|---|---|
| `memory/` | **индекс — каждую сессию** (always-in-context) | `MEMORY.md` = **роутер**: (1) поведенческие правила, (2) таблица маршрутизации `область/модуль → doc-хаб` (ось = модули), (3) атомарные гочи ≤3 строк | recall-ярус: «помни, что это существует» + куда идти за деталью. Стоит бюджета всегда → держим маленьким |
| `docs/conventions/` | по требованию (перед написанием такого кода) | code-shape конвенции (backend/frontend/db-migrations) | поведенческие правила — в `memory`; *code-shape* — здесь |
| `docs/` | **по требованию** (когда работаешь в этой области) | структурная проза: архитектура, контракты, runbook'и, модульные референсы, схемы БД | reference-ярус: глубина и длина бесплатны, пока не открыл |
| `tasks/` | индекс — каждую сессию | журнал работ: что сделано, проблемы, открытые хвосты, по файлу на область | хронология *действий*, а не знание. Отвечает «что уже делали», не «как устроено» |
| `plans/` | по запросу | планы будущей реализации (индекс + шаблон) | про *ещё не сделанное*; tasks — про факт сделанного |
| `research/` | перед исследовательской задачей | находки по **внешним** сервисам/API: поведение, лимиты, баги | чужое поведение; docs — про *наш* код и архитектуру |
| `tools/` | когда нужна повторяющаяся операция | shell-скрипты (рестарты бэкенда и т.п.) | исполняемое, а не описательное |
| `reference/` | по требованию | внешние справочные материалы (сырьё: спеки, OpenAPI) | чужой первоисточник; docs — наша осмысленная выжимка |

**Граница memory ↔ docs (single-home):** появились секции/проза → это **doc** (память держит максимум одну строку-указатель); одиночный факт 1–3 строки → только **memory**. Запись памяти, обросшая секциями, **поднимается в doc + указатель** — мини-доки в `memory/` не копим. Дублировать одну тему в обоих нельзя: docs владеет деталями, память — указателем. Подробно — раздел «Memory vs Docs» в `agent-primary.md`.

## Установка

Папка `AGENTS/` — настоящие файлы, коммитятся в git напрямую (отдельный снимок и скрипт синхронизации больше не нужны). В git не входят только корневые симлинки `AGENTS.md` и `CLAUDE.md` — их Claude Code ждёт в корне репозитория. После клона создай их:

```bash
# Симлинки на основной файл-инструкцию в корне
ln -s AGENTS/agent-primary.md CLAUDE.md
ln -s AGENTS/agent-primary.md AGENTS.md
```

## Советы и подводные камни

### Запуск команд: состояние шелла не сохраняется, venv не в PATH

Каждая команда агента стартует заново из профиля пользователя — **переменные окружения, `cd`, `source` и функции шелла между вызовами не сохраняются.** Активация venv в одном вызове (`source .venv/bin/activate`) к следующему вызову уже не действует.

Следствия:

- venv проекта **не в `PATH`**. Голый `pytest` / `alembic` / `ruff` падает с `No such file or directory`. Запускай инструменты venv через **`uv run <tool>`** (напр. `uv run pytest --core`) или по абсолютному пути (`.venv/bin/pytest`). `uv run` сам синхронизирует нужное окружение.
- Всё, что требует переменной окружения или рабочего каталога, должно быть **самодостаточным в одной команде**: задавай инлайн (`FOO=bar uv run ...`) или сцепляй через `&&` в том же вызове. Не полагайся на значение, экспортированное в предыдущем вызове.

### Память пишется в ~/.claude, а не в проект

Claude пишет авто-память в `~/.claude/projects/<encoded-project-path>/memory/` независимо от инструкций. Путь захардкожен — опции изменить его нет.

**Решение:** заменить системную папку памяти симлинком на папку памяти проекта:

```bash
PROJECT=/path/to/my-project
CLAUDE_MEM=~/.claude/projects/$(echo $PROJECT | sed 's|/|-|g' | sed 's|^-||')/memory

rm -rf "$CLAUDE_MEM"
ln -s "$PROJECT/AGENTS/memory" "$CLAUDE_MEM"
```

После этого вся авто-память пишется прямо в `AGENTS/memory/` — без дублирования и ручной синхронизации.

**Оговорка:** работает, только если оба пути на одной файловой системе. Если `~/.claude` и проект на разных точках монтирования (`df ~/.claude` vs `df /your/project`), Claude Code бросает `Invalid fd passed` при записи через симлинк. Тогда держи каталоги раздельно и синхронизируй вручную.
